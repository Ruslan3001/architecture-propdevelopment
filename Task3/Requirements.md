# Требования к безопасности внешних интеграций (Сервисы «Умный дом»)

Настоящий документ описывает архитектурные и технические требования ИБ для безопасной интеграции платформы PropDevelopment с внешними системами партнёра (управление интеллектуальными домофонами и шлагбаумами).

## 1. Организация взаимодействия между системами

Взаимодействие с внешней платформой должно строиться с соблюдением принципа единой точки входа и минимизации передаваемых данных:

* **Единый шлюз (API Gateway):** Все входящие и исходящие вызовы между мобильным приложением Собственника, внутренними системами PropDevelopment и платформой партнера маршрутизируются строго через централизованный API Gateway. Прямые обращения из мобильного приложения в обход шлюза запрещены.
* **Микросервис-адаптер (smart-home-app):** Внутри контура PropDevelopment выделяется отдельный сервис-посредник (Anti-Corruption Layer). Его задача — оркестрация запросов и полная анонимизация данных перед их отправкой партнёру.
* **Минимизация ПДн (Data Minimization):** Внешней платформе запрещено передавать явные персональные данные (ФИО, телефоны, email). Передаются только обезличенные идентификаторы (например, `tenant_id`, `device_id`), номера автомобилей и хеши/векторы биометрических данных (без передачи исходных фотографий).

## 2. Протоколы аутентификации и авторизации

Для обеспечения надежного контроля доступа на всех уровнях применяются современные стандарты:

* **Аутентификация конечных пользователей (Собственников):** 
  * Осуществляется через единый сервис `auth-service-1` (Keycloak).
  * **Протокол:** OpenID Connect (OIDC) поверх OAuth 2.0.
  * Мобильное приложение использует короткоживущие Access-токены (JWT) и Refresh-токены для поддержания сессии.
* **Межсервисная B2B интеграция (PropDevelopment ↔ Партнёр):**
  * **Протокол транспортного уровня:** Строгое применение **mTLS** (Mutual TLS) для двусторонней проверки сертификатов серверов, что гарантирует защиту от подмены участника взаимодействия (Man-in-the-Middle).
  * **Протокол прикладного уровня:** OAuth 2.0 (Client Credentials Grant) для безопасного обмена технологическими учетными данными и получения токенов авторизации API.
* **Авторизация и защита от BOLA/IDOR:**
  * Применяется атрибутный контроль доступа (**ABAC**). Микросервис `smart-home-app` перед выполнением команды проверяет по базе `smart-home-db`, имеет ли пользователь с данным JWT-токеном права на открытие конкретного домофона или шлагбаума.

## 3. Требования к информационной безопасности (ИБ)

* **Конфиденциальность и целостность транзитных данных:**
  * Любой сетевой трафик, покидающий защищенный периметр, должен быть зашифрован с использованием протокола TLS версии не ниже 1.2 (рекомендуется TLS 1.3) и современных шифрсьютов (Cipher Suites).
* **Защита доступности (Availability):**
  * На уровне API Gateway настраиваются строгие лимиты частоты запросов (Rate Limiting и Throttling), чтобы защитить партнёрскую платформу и внутренние сервисы от DDoS-атак и флуда.
  * Витрины защищаются с помощью WAF (Web Application Firewall).
* **Аудит и логирование (Auditing):**
  * Все попытки открытия дверей/шлагбаумов, ошибки авторизации, а также административные действия с правами доступа должны журналироваться.
  * Логи должны автоматически агрегироваться и отправляться в централизованную систему мониторинга инцидентов (SIEM) по протоколу Syslog без содержания чувствительных ПДн.
* **Соблюдение законодательства (ФЗ-152):**
  * Сбор и передача биометрических данных и номерных знаков автомобилей осуществляются исключительно после подписания Собственником Согласия на обработку ПДн (в электронной форме через мобильное приложение).
  * Биометрические данные должны храниться в зашифрованном виде (TDE на уровне базы данных).
